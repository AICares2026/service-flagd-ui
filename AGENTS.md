## Stack
JavaScript (Node.js); flagd UI service; Docker-based deployment.

## Constraints
Never modify:
- `package-lock.json` (regenerated by npm, not hand-edited)
- `Dockerfile`, `docker-compose.yml`, or any `*.dockerfile`
- Any `*.min.js` or bundled/generated output files
- `.env`, `.env.*`, credential or secret files
- Test fixture files and snapshot files

## Conventions
- JavaScript source lives in `src/` (assumed); tests co-located or in `test/` or `__tests__/`
- Test files follow `*.test.js` or `*.spec.js` naming
- npm scripts defined in `package.json` — run `npm test` to execute the test suite
- Docker is the primary runtime; local dev uses npm

## Dependency manifests
- `package.json` — all npm dependency versions declared here
- `package-lock.json` — lockfile, never edit directly; regenerate with `npm install` after changing `package.json`

## Agent rules
- When updating npm dependencies, edit only `package.json` version ranges, then run `npm install` to regenerate the lockfile
- Do not upgrade major versions without checking for breaking changes in the changelog
- CVE scan agent (`cve_scan`) has consistently produced 0 file changes — verify scan tooling is functional before concluding no CVEs exist
- When adding tests, match existing naming conventions and place files alongside source or in the established test directory
- Never introduce new runtime dependencies without a corresponding entry in `package.json`
